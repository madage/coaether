# Message Bus

Plugins can communicate asynchronously with other plugins and AI agents via the message bus.

## Message Format

All messages use the JSON envelope format:

```json
{
  "from": "plugin:my-plugin",
  "to": "broadcast",
  "type": "plugin.my-plugin:event.widget_created",
  "payload": {
    "widget_id": "w123",
    "name": "My Widget"
  },
  "id": "a1b2c3d4-...",
  "timestamp": 1717000000000
}
```

| Field | Description |
|-------|-------------|
| `from` | Sender, format `plugin:{name}` |
| `to` | Target, see valid values below |
| `type` | Message type, see naming conventions |
| `payload` | Message content (any JSON) |
| `id` | Unique ID (generated by host) |
| `timestamp` | Unix timestamp in milliseconds |

## Sending Messages

Sent via the [Host API](04-host-api.md):

```
POST /__plugin_host/message
X-Plugin-Id: my-plugin
Content-Type: application/json

{
  "to": "broadcast",
  "type": "plugin.my-plugin:event.widget_created",
  "payload": {"widget_id": "w123"}
}
```

## Message Targets

| to value | Description |
|----------|-------------|
| `broadcast` | Broadcast to all connected endpoints and plugins |
| `session://{sessionID}` | Send to all members of a specific session |
| `plugin:{name}` | Send to a specific plugin |
| `agent://{nodeId}/{agentId}` | Send to a specific AI agent |

## Message Type Naming Convention

```
plugin.{name}:{category}.{action}
```

| Category | Naming | Description |
|----------|--------|-------------|
| Event | `plugin.my-plugin:event.widget_created` | Notification event |
| Command | `plugin.my-plugin:command.sync_data` | Request the recipient to perform an action |
| Data | `plugin.my-plugin:data.widget_updated` | Data synchronization |
| Request | `plugin.my-plugin:request.summarize` | Request expecting a reply |

### Wildcard Matching

Plugins can declare interest in specific types via `capabilities.message_types`. Glob patterns are supported:

```json
{
  "capabilities": {
    "message_types": ["plugin.my-plugin:event.*"]
  }
}
```

## Communication Patterns

### Event Pattern (Fire-and-Forget)

```
PluginA                          PluginB
  │                                │
  │── event.widget_created ───────→│
  │                                │  (no reply expected)
```

### Request-Response Pattern

```
PluginA                          PluginB
  │                                │
  │── request.summarize ─────────→│
  │                                │
  │←─ response.summarize ─────────│
  │    (correlation id: use same message ID)  │
```

### Command Pattern

```
PluginA                          PluginB
  │                                │
  │── command.sync_data ─────────→│
  │    to: "plugin:target-plugin"  │
  │                                │  (explicit target specified)
```

## Receiving Messages

Plugins receive message bus messages via `POST /__plugin/hook` (triggered when a message arrives, firing the `plugin:message` hook):

```json
{
  "hook_name": "plugin:message",
  "context": {
    "from": "plugin:other-plugin",
    "type": "plugin.other-plugin:event.something",
    "payload": "{\"key\": \"value\"}"
  },
  "async": true
}
```

## Complete Go Plugin Example

```go
package main

import (
    "bytes"
    "encoding/json"
    "net/http"
    "os"
)

func sendEvent(eventType string, payload interface{}) {
    hostAddr := os.Getenv("COAETHER_HOST_ADDR")
    pluginID := os.Getenv("COAETHER_PLUGIN_ID")

    body, _ := json.Marshal(map[string]interface{}{
        "to":      "broadcast",
        "type":    "plugin." + pluginID + ":event." + eventType,
        "payload": payload,
    })

    req, _ := http.NewRequest("POST",
        "http://" + hostAddr + "/__plugin_host/message",
        bytes.NewReader(body))
    req.Header.Set("X-Plugin-Id", pluginID)
    req.Header.Set("Content-Type", "application/json")

    http.DefaultClient.Do(req)
}

// Usage example
func notifyWidgetCreated(widgetID string) {
    sendEvent("widget_created", map[string]string{
        "widget_id": widgetID,
    })
}
```

## Important Notes

1. **Message persistence**: The message bus does not guarantee message delivery by default; messages will be lost if the target is offline
2. **Message size**: Keep payload under 1MB; for large messages, use file storage
3. **Avoid excessive broadcasting**: Broadcast messages are sent to all endpoints and plugins; frequent broadcasting impacts performance
4. **Unique message types**: Each plugin uses its own namespace to avoid conflicts with other plugins
